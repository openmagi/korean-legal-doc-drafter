# 용역계약서(IT 및 소프트웨어 — 정보화사업) (doc-198)

> 카테고리: 용역계약 | 난이도: 상 | 예상 질문 라운드: 6회

## 용도와 법적 성격

- 소프트웨어·웹/앱·시스템 개발, 유지보수, 데이터 구축 등 IT 용역을 외주할 때 쓰는 개발 도급계약서다. 제작물(결과물)이 있는 개발형과 제작물 없는 운영·유지보수형을 모두 커버한다.
- 개발형은 민법상 **도급**(제664조)으로 일의 완성 의무·하자담보책임이 핵심이고, 운영·유지보수형은 **위임** 성격이 강하다. 소프트웨어진흥법은 공공 SW사업에 적용되며 민간 계약에는 직접 적용되지 않으나 하자담보·검수 관행의 참고 기준이 된다.
- 유사 문서와의 구분: 일반 용역계약서(doc-037)와 달리 **지식재산권 양도 구조, 소스코드 인도, 단계별 검수, 하자보수 기간**이 표준 장비로 들어간다. SaaS 구독·라이선스 계약(이용권 부여)과는 권리 이전 구조가 다르다.

## 작성 전 확인사항

- **저작권 함정**: 저작권법 제9조(업무상저작물)는 "법인 등의 기획 하에 그 업무에 종사하는 자가 작성"한 경우에만 법인 귀속을 인정한다 — **외주 개발사(도급)는 "업무에 종사하는 자"가 아니므로 제9조가 적용되지 않고, 저작권은 개발사(을)에 원시 귀속**한다. 발주사(갑)가 권리를 가지려면 계약서에 명시적 양도 조항이 반드시 필요하다. 이를 빠뜨린 채 대금만 지급한 발주사가 소스코드를 받지 못하거나 2차 개발을 못 하는 분쟁이 매우 흔하다.
- 요구사항이 확정되지 않은 상태의 "일단 계약"은 범위 분쟁(scope creep)의 원인이다. 요구사항정의서/기능명세서를 별지로 첨부할 수 있는지 확인.
- 개발사가 오픈소스를 쓸 가능성이 높으므로 오픈소스 라이선스 고지·보증 조항을 검토.
- 발주자가 원사업자(중견·대기업 등)이고 수급사업자가 중소기업이면 하도급법이 적용될 수 있다(대금 60일 내 지급, 서면 발급 의무 등 — 최신 기준 확인 필요).
- 준비물: 요구사항 문서(또는 RFP), 일정 계획(중간 산출물 포함), 검수 기준 초안, 운영 환경 정보.

## 정보 수집 단계

### 1단계 — 당사자와 사업 유형
- **질문**: 갑(발주)·을(개발) 각각의 형태는? (① 법인 ② 개인사업자 ③ 개인) 이 사업은 ① 결과물을 만들어 인도하는 개발형인가요 ② 제작물 없이 운영·유지보수·기술지원을 제공하는 수행형인가요?
- **분기**: ①(제작물 있음)이면 검수→지재권 양도→하자보수의 풀 구조를 쓴다. ②(제작물 없음)이면 검수·하자보수 조항을 서비스수준(SLA)·월간 보고로 대체하고 지재권 조항은 산출 문서 한정으로 축소한다.
- **주의**: 을이 개인 프리랜서 개발자이고 갑 사무실 상주·상시 지휘를 받는 구조면 위장도급(근로자성) 리스크를 경고한다.

### 2단계 — 개발 범위와 일정
- **질문**: 개발 대상(시스템/앱/기능)과 범위를 알려주세요. 요구사항 문서가 있나요? 단계(분석-설계-개발-테스트-오픈) 일정과 중간 산출물은?
- **분기**: 요구사항 문서가 있으면 "별지 1 요구사항정의서" 첨부 방식으로, 없으면 핵심 기능 목록을 계약서 본문에 열거하고 "상세 요구사항은 착수 후 [주]주 내 상호 확정" 절차를 넣는다. 변경 요청은 변경관리 조항(서면 합의 + 대금·일정 조정)으로 처리.
- **주의**: "갑의 요청 사항 반영"류의 포괄 문구는 무한 수정의 근거가 된다. 무상 수정 범위(검수 전 경미한 수정)와 유상 변경(기능 추가·요건 변경)의 경계를 명시하라.

### 3단계 — 대금과 지급 구조
- **질문**: 총 대금과 지급 방식은? ① 일시금 ② 계약금+잔금 ③ 계약금+중도금+잔금. 잔금의 지급 조건(검수 합격 후 N일)은? 추가 비용(서버·라이선스·외부 API 등)은 누가 부담하나요? 성공보수(오픈 후 성과 연동 등) 약정이 있나요?
- **분기**: ③이면 중도금 지급 시점을 중간 산출물 승인과 연동한다("설계 산출물 승인 시 중도금"). 추가 비용 갑 부담 시 사전 승인 절차, 을 부담 시 대금에 포함됨을 명시.
- **주의**: 잔금을 "오픈 후" 또는 "검수 후"로만 쓰면 갑이 검수·오픈을 미루는 만큼 잔금이 밀린다. 간주검수(인도 후 [14]일 내 이의 없으면 합격) 조항이 을의 생명줄이다. 반대로 갑은 검수 기준 미달 시 보완 요구권과 기한을 확보해야 한다.

### 4단계 — 검수 기준과 절차
- **질문**: 검수는 무엇으로 판정하나요? (요구사항정의서 기준 / 별도 검수 시나리오 / 테스트 케이스 통과율 등) 검수 기간과 보완 횟수·기한은?
- **분기**: 객관 기준(테스트 케이스, 기능 목록 체크)이 있으면 그것을 검수 기준으로 명시. 없으면 "별지 요구사항정의서에 명시된 기능의 정상 동작"을 기준으로. 보완 후 재검수는 [회] 이내·회당 [일]일로 상한을 둔다.
- **주의**: 검수 기준 없는 "갑이 만족할 때까지"는 사실상 무한 책임이다. 하자(기준 미달)와 추가 요구(기준 외 신규 요건)를 구분하는 문장을 반드시 넣는다.

### 5단계 — 지식재산권·소스코드·보증
- **질문**: ⑴ 결과물의 저작권 등 지식재산권을 갑에게 양도하나요, 을이 보유하고 갑은 사용권만 갖나요? ⑵ 소스코드(주석·빌드 스크립트·문서 포함)를 인도하나요? ⑶ 을이 제3자 권리 비침해를 보증하나요? ⑷ 오픈소스 사용 고지 의무를 둘까요?
- **분기**: 양도형이면 "잔금 완납 시 2차적저작물작성권 포함 양도"(저작권법 제45조 제2항 — 2차적저작물작성권은 명시 없으면 유보 추정). 을 보유+사용권형이면 사용권의 범위(기간·독점 여부·수정 가능 여부)를 정밀하게 받는다. 을의 기존 보유 모듈(배경 IP)은 양도 대상에서 제외하고 갑에게 사용권만 부여하는 분리 구조를 제안.
- **주의**: 소스코드 인도를 명시하지 않으면 실행파일만 받고 끝나는 수가 있다. 인도물 목록(소스코드 저장소 전체, DB 스키마, 배포 스크립트, 관리자 매뉴얼)을 별지화하라. 오픈소스는 GPL 계열 포함 시 갑의 소스 공개 의무가 생길 수 있으므로 사용 목록 고지 의무가 실무상 중요하다.

### 6단계 — 하자보수·해지·기타 의무
- **질문**: ⑴ 검수 합격 후 무상 하자보수 기간은? (통상 3개월~1년) ⑵ 보고 방식은? (① 규정 안 함 ② 갑 요청 시 ③ 정기 보고) ⑶ 계약 종료 사유는? (① 특정 날짜 ② 검수 완료 등 특정 조건 ③ 자동 연장 — 유지보수형) ⑷ 추가 의무 선택: 비밀유지 / 경쟁업체 관여 금지 / 위반 시 위약금
- **분기**: 하자보수는 "검수 합격일부터 [개월]개월간 요구사항 기준 미달 결함 무상 보수, 단 갑의 환경 변경·임의 수정으로 인한 것 제외"로. 유지보수형 계약이면 하자보수 대신 SLA(응답 시간·복구 시간)를 쓴다. 위약금을 넣으면 손해배상액의 예정(법원 감액 가능)인지 위약벌(별도 배상 청구 가능, 과다 시 일부 무효)인지 성격을 정해 준다.
- **주의**: 하자담보책임의 법정 기간과 별도로 계약상 무상 보수 기간을 정하는 것이므로, "무상 보수 종료 후에도 법률상 하자담보책임은 영향받지 않는다"는 확인 문구가 분쟁 예방에 좋다.

## 필수 기재사항 체크리스트

- [ ] 개발 범위·요구사항(별지 첨부) — 누락 시 scope creep과 검수 분쟁
- [ ] 변경관리 절차(서면 합의 + 대금·일정 조정) — 누락 시 무상 추가 개발 강요
- [ ] 대금·지급 단계·간주검수 — 누락 시 잔금 무기한 지연
- [ ] 검수 기준·보완 횟수 상한 — 누락 시 "만족할 때까지" 무한 수정
- [ ] **지식재산권 명시적 양도(2차적저작물작성권 포함) 또는 사용권 범위** — 저작권법 §9 부적용으로 묵시 이전 없음
- [ ] 소스코드 등 인도물 목록 — 누락 시 실행파일만 인도받는 위험
- [ ] 제3자 권리 비침해 보증·오픈소스 고지 — 누락 시 갑이 침해 분쟁 떠안음
- [ ] 무상 하자보수 기간과 면책 사유 — 누락 시 하자 범위 다툼
- [ ] 해제·해지와 기성 정산 — 누락 시 중도 종료 시 산출물·대금 공중분해
- [ ] 비밀유지·개인정보 처리(수탁 개발 시 개인정보보호법 §26 위탁 문서화)
- [ ] 작성일·서명란·각 1부 보관
